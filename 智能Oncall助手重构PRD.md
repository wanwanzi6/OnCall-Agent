# 智能 Oncall 助手重构 PRD

## 1. 背景

当前项目已经具备智能对话、知识库上传、RAG 检索、Agent 工具调用、AI Ops 分析入口等原型能力，但整体仍偏 PoC：

- 工程结构不够清晰，控制器、Agent、工具、基础设施逻辑耦合较多。
- 配置、密钥、路径、端口存在硬编码问题。
- 告警、日志、数据库工具的安全边界不足。
- AI Ops 链路尚未形成稳定、可演示、可测试的完整闭环。
- 缺少明确的产品目标、MVP 范围和验收标准。

本次重构目标是将项目升级为一个适合作为简历项目和技术展示的智能 Oncall/AIOps Agent 系统。

## 2. 产品定位

智能 Oncall 助手是一个面向研发、运维、SRE 的故障排查辅助系统。

系统通过内部知识库、告警数据、日志数据、指标数据和大模型 Agent 编排能力，辅助 Oncall 工程师完成告警理解、SOP 匹配、证据查询、根因分析和故障报告生成。

一句话定位：

> 基于 RAG 和多步 Agent 工具调用的智能 Oncall 故障分析助手。

## 3. 目标用户

### 3.1 主要用户

- 后端研发工程师
- SRE/运维工程师
- 平台工程师
- 一线 Oncall 值班人员

### 3.2 用户痛点

- 告警发生后，需要人工查找对应 SOP 和历史处理手册。
- 日志、指标、告警平台分散，排查链路长。
- 新人 Oncall 对系统背景和常见故障不熟悉。
- 故障分析过程缺少结构化沉淀。
- 复杂告警需要多轮查询和判断，人工容易遗漏关键证据。

## 4. 产品目标

### 4.1 MVP 目标

第一阶段重构后的 MVP 需要支持一条完整可演示链路：

1. 用户上传告警处理手册或运维 SOP。
2. 系统解析文档、切片、向量化并写入知识库。
3. 用户发起普通问答，系统基于知识库进行 RAG 回答。
4. 用户点击 AI Ops 分析入口。
5. 系统查询活跃告警。
6. 系统根据告警名称检索对应 SOP。
7. 系统调用日志或指标工具收集证据。
8. 系统生成结构化告警分析报告。
9. 用户可以查看 Agent 执行步骤和最终分析结论。

### 4.2 简历展示目标

项目应能清晰体现以下能力：

- Go 后端工程分层设计。
- RAG 知识库构建与检索。
- Milvus 向量数据库使用。
- LLM Agent 工具调用。
- 多步故障分析工作流。
- SSE 流式输出。
- 安全配置管理和工具调用边界控制。
- Docker Compose 本地可演示部署。

## 5. 核心使用场景

### 5.1 知识库构建

用户上传内部告警处理手册、运维 SOP 或故障复盘文档。

系统处理流程：

1. 校验文件类型和大小。
2. 保存原始文档。
3. 解析文档内容。
4. 按标题和长度进行切片。
5. 调用 embedding 模型生成向量。
6. 将 chunk、元数据和向量写入 Milvus。
7. 返回索引状态。

### 5.2 智能问答

用户以自然语言询问告警、错误码、处理步骤或系统背景。

系统处理流程：

1. 接收用户问题。
2. 根据问题检索知识库。
3. 将检索结果、会话上下文和用户问题组合为 prompt。
4. 调用大模型生成回答。
5. 返回答案和引用来源。

### 5.3 AI Ops 自动分析

用户点击 AI Ops 分析按钮，系统自动分析当前活跃告警。

系统处理流程：

1. 查询活跃告警。
2. 提取告警名称、服务名、实例、地域、触发时间等信息。
3. 根据告警名称检索对应 SOP。
4. 根据 SOP 中的排查要求调用日志或指标工具。
5. 汇总告警、SOP、日志、指标证据。
6. 生成结构化故障分析报告。
7. 展示 Agent 执行步骤和最终报告。

### 5.4 历史报告查看

用户可以查看历史 AI Ops 分析报告。

系统处理流程：

1. 展示历史分析记录列表。
2. 用户选择某次分析记录。
3. 展示告警信息、执行步骤、证据摘要和最终报告。

## 6. MVP 功能范围

### 6.1 必须实现

- 普通聊天问答。
- SSE 流式聊天。
- Markdown/TXT 文档上传。
- 文档切片与向量索引。
- 基于 Milvus 的知识库检索。
- RAG 增强问答。
- Mock 告警数据源。
- Mock 日志数据源。
- AI Ops 一键分析。
- 结构化告警分析报告。
- Agent 执行步骤记录。
- 基础配置管理。
- Docker Compose 本地启动。

### 6.2 优先实现

- 文档列表查询。
- 删除知识库文档。
- 重建知识库索引。
- 报告历史记录。
- 工具调用超时控制。
- 统一错误返回。
- 基础单元测试。
- README demo 指南。

### 6.3 暂不实现

- 多租户。
- 复杂 RBAC 权限系统。
- 任意 SQL 执行。
- 自动执行修复动作。
- 生产级告警降噪。
- 多模型动态路由。
- PDF/DOCX 全格式解析。
- Kubernetes 生产部署。

## 7. 非目标

本阶段重构不追求直接替代真实生产 Oncall 平台。

以下能力不作为阶段 0 到 MVP 的目标：

- 自动修改线上配置。
- 自动重启服务。
- 自动执行数据库变更。
- 自动关闭告警。
- 对所有故障类型做准确根因判断。
- 完整替代 Prometheus、日志平台、工单系统。

系统定位是“辅助分析和决策”，不是“自动处置系统”。

## 8. 核心模块设计

### 8.1 前端模块

- Chat 页面：普通问答和流式问答。
- Knowledge 页面：文档上传、文档列表、索引状态。
- AI Ops 页面：触发告警分析、展示执行步骤和报告。
- Reports 页面：历史分析报告列表和详情。

### 8.2 后端模块

建议重构后的后端模块：


### 8.3 Agent 工作流

AI Ops Agent 建议拆成以下节点：

```text
AlertCollector
SOPRetriever
EvidenceCollector
RootCauseAnalyzer
ReportGenerator
```

节点职责：

- AlertCollector：获取活跃告警。
- SOPRetriever：根据告警检索处理手册。
- EvidenceCollector：根据 SOP 查询日志或指标。
- RootCauseAnalyzer：结合证据分析可能根因。
- ReportGenerator：生成结构化报告。

## 9. 工具设计

### 9.1 告警工具

工具名：`query_active_alerts`

能力：

- 查询当前活跃告警。
- 支持 mock 模式。
- 后续支持 Prometheus 模式。

输入：

- 可选的服务名、告警名、严重级别过滤条件。

输出：

- 告警名称。
- 服务名。
- 严重级别。
- 触发时间。
- 描述。
- 标签信息。

### 9.2 日志工具

工具名：`query_logs`

能力：

- 根据关键词、地域、日志主题、时间范围查询日志。
- 支持 mock 模式。
- 后续支持真实日志平台。

输入：

- region。
- topic_id。
- keyword。
- start_time。
- end_time。

输出：

- 日志摘要。
- 命中数量。
- 关键错误样例。

### 9.3 文档检索工具

工具名：`query_internal_docs`

能力：

- 根据问题或告警名称检索内部知识库。

输入：

- query。
- top_k。

输出：

- 文档片段。
- 来源文档。
- 相似度分数。

### 9.4 时间工具

工具名：`get_current_time`

能力：

- 获取当前时间。
- 为日志和指标查询提供时间参数。

## 11. 安全要求

### 11.1 配置安全

- 不允许提交真实 API Key。
- 仓库只保留 `.env.example`。
- 所有敏感配置从环境变量读取。

### 11.2 工具安全

- 禁止模型执行任意 SQL。
- 禁止模型直接执行系统命令。
- 工具必须有输入 schema。
- 工具必须有超时。
- 工具失败只能返回错误，不能终止进程。
- 工具调用需要记录 trace。

### 11.3 文件上传安全

- 限制文件类型。
- 限制文件大小。
- 清理文件名。
- 防止路径穿越。
- 上传目录配置化。

## 12. 可观测性要求

每次 AI Ops 分析需要生成 trace id。

需要记录：

- 用户请求。
- Agent 执行步骤。
- 工具调用名称。
- 工具输入摘要。
- 工具输出摘要。
- 错误信息。
- 最终报告。

## 13. 验收标准

### 13.1 MVP 验收

- 能通过 Docker Compose 启动后端、前端和 Milvus。
- 能上传 Markdown/TXT 文档并完成索引。
- 能基于上传文档完成 RAG 问答。
- 能点击 AI Ops 触发 mock 告警分析。
- 能生成结构化告警分析报告。
- 能看到 Agent 执行步骤。
- 项目中没有真实 API Key。
- 工具错误不会导致服务进程退出。
- README 能指导用户完成一次 demo。

### 13.2 技术验收

- `go test ./...` 可通过。
- 核心模块有基础单元测试。
- 配置项有 `.env.example`。
- 日志中包含 trace id。
- 主要外部调用有超时控制。

## 14. Demo 脚本

推荐演示流程：

1. 启动项目。
2. 上传 `告警处理手册.md`。
3. 提问：“服务下线告警应该怎么处理？”
4. 系统返回基于 SOP 的处理步骤。
5. 点击 AI Ops。
6. 系统读取 mock 活跃告警。
7. 系统匹配 “服务下线” SOP。
8. 系统查询 mock 日志。
9. 系统输出告警分析报告。
10. 展示执行步骤和证据链。
